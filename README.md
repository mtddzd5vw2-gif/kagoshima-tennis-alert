# kagoshima-tennis-alert

鹿児島県のテニスコート予約サイトを確認し、指定時間帯に空き候補が見つかった場合に LINE Messaging API で通知するためのプロジェクトです。

> [!WARNING]
> 現在は初期検証段階です。監視対象日が `2026-05-03` に固定されており、定期実行も無効です。そのため、このままでは継続的な空き状況監視には利用できません。

## 現在の動作

GitHub Actions の手動実行により、次の処理を行います。

1. Python 3.11 と Playwright/Chromium をセットアップする
2. LINE に疎通確認用のテストメッセージを送信する
3. 鴨池県営テニスコートの予約ページを開く
4. 8:00〜13:00 の空き候補をページ内の文言から抽出する
5. 候補が見つかった場合に LINE へ通知する
6. 調査用の HTML とスクリーンショットを Actions のArtifactとして3日間保存する

空き枠の予約処理は行いません。予約は必ず公式サイトで確認のうえ、利用者自身で行ってください。

## セットアップ

### 1. LINE Messaging APIを準備する

LINE DevelopersでMessaging APIチャネルを作成し、通知先ユーザーが公式アカウントを友だち追加していることを確認します。

### 2. GitHub Actions Secretsを登録する

GitHubリポジトリの `Settings` → `Secrets and variables` → `Actions` で、次のRepository secretsを登録します。

| Secret | 用途 |
| --- | --- |
| `LINE_CHANNEL_ACCESS_TOKEN` | LINE Messaging APIのチャネルアクセストークン |
| `LINE_USER_ID` | Push messageの通知先ユーザーID |

値はログやコードに直接記載しないでください。どちらかが未設定の場合、現在のワークフローはテストメッセージ送信時に失敗します。

## 手動実行

1. GitHubリポジトリの `Actions` を開く
2. `Check tennis court availability` を選択する
3. `Run workflow` を選択して実行する
4. 実行ログと `spm-page-snapshots` Artifactを確認する

現在は `workflow_dispatch` のみが設定されているため、自動実行されません。また、実行するたびに空き状況とは無関係なLINEテストメッセージが1件送信されます。

## 現在の固定設定

設定値は [`.github/workflows/check-tennis.yml`](.github/workflows/check-tennis.yml) 内に直接記述されています。

| 項目 | 現在値 |
| --- | --- |
| 対象施設 | 鴨池県営テニスコート |
| 対象日 | 2026-05-03 |
| 対象時間 | 8:00〜13:00 |
| 予約ページ | `v2.spm-cloud.com` の日別予約ページ |
| 実行方法 | GitHub Actionsからの手動実行 |
| Artifact保持期間 | 3日 |

## 既知の課題

- 対象日が過去の日付に固定されており、日常運用できない
- 定期実行が無効で、手動実行しかできない
- 実行のたびにLINEテストメッセージが送信される
- 監視処理がワークフローへ埋め込まれ、ローカル実行・保守・テストが難しい
- Playwrightのバージョンが固定されておらず、実行結果の再現性が低い
- ページ内文言を広く探索するヒューリスティック方式のため、誤検知・見逃しの可能性がある
- 同じ空き候補を何度も通知しないための状態管理がない
- lint、単体テスト、ワークフロー検証がない
- `actions: write` 権限が設定されており、必要最小限の権限か再確認が必要
- GitHub Actionsの実行履歴・成功状況はこのREADME更新時点では未確認

## 改善ロードマップ

### Phase 1: 実行可能なプロジェクトへ分離

- 監視処理を `src/` 配下のPythonモジュールへ移す
- 依存関係をバージョン固定したファイルで管理する
- 施設、対象日、時間帯を環境変数または設定ファイルから変更できるようにする
- LINE疎通確認を通常の監視処理から分離する

### Phase 2: 品質と安全性を高める

- 時刻判定、空き判定、通知文生成の単体テストを追加する
- Ruffなどの静的解析とテストをGitHub Actionsで実行する
- ワークフロー権限を必要最小限にする
- 外部ActionとPython依存関係のバージョンを固定する
- 通信失敗時のリトライ、タイムアウト、エラー通知を整備する

### Phase 3: 継続監視を実装する

- 対象日を当日からの相対日付などで自動計算する
- GitHub ActionsのscheduleをJST運用時間に合わせて有効化する
- 通知済み候補を記録し、重複通知を抑止する
- 複数施設・複数日・複数時間帯へ拡張する
- 予約サイトのHTML変更を検知できる監視を追加する

## 注意事項

- 予約サイトの利用規約、アクセス頻度、robots設定を確認して運用してください。
- 短い間隔でのアクセスや過剰な並列実行は避けてください。
- 予約サイトの仕様変更により、予告なく監視が動作しなくなる可能性があります。
- Artifactには取得ページの内容が含まれます。公開範囲と保持期間を適切に管理してください。
